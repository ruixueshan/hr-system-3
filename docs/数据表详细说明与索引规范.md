# HR系统3.0 — 数据表详细说明与索引规范

> 版本：v2.0（2026-05-15）
> 基于：collections.json、DATABASE_MODEL_BOUNDARIES.md、INDEXES.md、生产库实测
> 环境：cloud1-5glojms9a83c3457

---

## 说明

- ⭐ = 该域核心表
- 🎯 = 多表关联的枢纽表
- 索引标记：`必加`= 线上必须存在的索引，`建议`= 性能优化推荐
- 字段仅列出关键业务字段，省略通用审计字段（`created_at`、`updated_at`、`_id`）

---

## 🔐 一、用户与认证

---

### 1. users ⭐ — 统一账号表

**数据来源**：小程序登录 / 扫码报名 / 后台创建

**定位**：小程序和 Web 端的登录账号，不保存员工实名主数据。

**写入入口**：
- 扫码报名 `submitJobApply()` → 新建
- 面试签到 `ensureCandidateUser()` → 新建
- 小程序登录 `auth-login` → 新建/更新
- 后台管理页 → 手动创建

| 字段 | 类型 | 说明 |
|------|------|------|
| `openid` | string | 微信小程序登录标识（唯一） |
| `unionid` | string | 微信开放平台统一标识 |
| `phone` | string | 登录手机号 |
| `account_phone` | string | 账号手机号（兼容字段） |
| `name` | string | 昵称或展示名 |
| `real_name` | string | 实名快照（兼容字段，新规范用 `real_name_snapshot`） |
| `avatar` | string | 头像 |
| `user_type` | string | `candidate` / `employee` / `admin` |
| `role` | string | 后台角色：`gm` / `deputy` / `hr` / `external` / `finance` |
| `status` | string | `normal` / `disabled` |
| `employee_id` | string | **绑定的员工主档 ID** → `employees._id` |
| `employee_no` | string | 员工编号快照（兼容） |
| `last_login` | date | 最近登录 |

**禁止写入**：`id_card`、`company_id`、`job_id`、`referrer_id`、`referrer_name`

**索引：**

| 索引字段 | 类型 | 用途 |
|----------|------|------|
| `openid` | 必加 | 微信小程序登录查询（最核心） |
| `phone` | 必加 | 手机号登录/查重 |
| `employee_id` | 必加 | 员工绑定查询（从 employee 反向查） |
| `user_type` + `status` | 建议 | 后台用户列表筛选 |

### 2. roles — 角色表

**数据来源**：后台配置

**定位**：角色权限定义，控制 Web 端和后台的菜单与操作权限。

**写入入口**：后台权限管理页

| 字段 | 说明 |
|------|------|
| `name` | 角色名称 |
| `status` | 状态 |
| `permissions` | 权限列表 |

**索引：**

| 索引字段 | 类型 | 用途 |
|----------|------|------|
| `name` | 建议 | 按名称查询 |

---

### 3. login_tokens — 登录令牌

**数据来源**：用户登录时自动生成

**定位**：存储登录态 token，用于接口鉴权。登录时写入，登出或过期后删除。

**写入入口**：
- `auth-login` / `auth-phone-login` → 登录时写入
- `auth` → token 校验

| 字段 | 说明 |
|------|------|
| `user_id` | → `users._id` |
| `token` | 登录令牌 |
| `expire_at` | 过期时间 |
| `created_at` | 创建时间 |

**索引：**

| 索引字段 | 类型 | 用途 |
|----------|------|------|
| `token` | 必加 | 按令牌查询（每次接口鉴权） |
| `user_id` | 建议 | 用户所有 token 查询 |

---

### 4. verification_codes — 验证码

**数据来源**：用户请求短信验证码时

**定位**：手机验证码记录，用于登录/注册验证，通常有 5 分钟有效期。

**写入入口**：
- `auth-login` → 发送验证码时写入

| 字段 | 说明 |
|------|------|
| `phone` | 手机号 |
| `code` | 验证码 |
| `expire_at` | 过期时间 |
| `used` | 是否已使用 |

**索引：**

| 索引字段 | 类型 | 用途 |
|----------|------|------|
| `phone` + `created_at` | 必加 | 按手机号查最近验证码 |

| `salary_type` | 建议 | 按薪资类型筛选 |
| `sort_order` | 建议 | 排序展示 |

### 7. qr_codes — 报名二维码

**数据来源**：后台生成

**定位**：每个二维码绑定一个岗位和推荐人，候选人扫码后自动带出推荐关系。

**写入入口**：
- Web 后台扫码报名二维码生成 → `qrcode` 云函数

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | string | **二维码唯一编码** |
| `type` | string | `job_apply`（报名） / `internal_onboard`（内部入职） / `interview_checkin`（面试签到） / `agent_referral`（代理） |
| `job_id` | string | → `jobs._id` |
| `job_name` | string | 岗位名称快照 |
| `company_id` | string | → `companies._id` |
| `company_name` | string | 企业名称快照 |
| `recommender_id` | string | **推荐人 ID** → `users._id` |
| `recommender_name` | string | 推荐人名称快照 |
| `creator_id` | string | 创建人 |
| `qr_url` | string | 小程序码图片 URL |
| `max_uses` | number | 最大使用次数 |
| `used_count` | number | 已使用次数 |
| `scan_count` | number | 扫码次数 |
| `expire_time` | date | 过期时间 |
| `status` | string | `active` / `paused` / `expired` |
| `internal_only` | boolean | 是否仅限内部使用 |
| `supports_daily` | boolean | 是否支持日结 |
| `interview_time` | string | 面试时间（报名码专用） |
| `interview_date` | date | 面试日期（签到码专用） |
| `landing_page` | string | 扫码后跳转页面 |

**索引：**

| 索引字段 | 类型 | 用途 |
|----------|------|------|
| `code` | 必加（唯一） | 扫码查询（最核心，每次扫码） |
| `job_id` + `status` | 必加 | 岗位下的二维码列表 |
| `recommender_id` | 建议 | 按推荐人查二维码 |
| `type` | 建议 | 按类型筛选 |
| `created_at` | 建议 | 按创建时间排序 |

## 📝 三、报名与面试

---

### 8. applications ⭐ — 报名投递

**数据来源**：扫码报名 / 小程序报名 / 后台代报名

**定位**：候选人投递岗位的唯一入口。**推荐人在此记录，用于日报"今日报名"统计。**

**写入入口**：
- `qrcode.job-apply` → 扫码报名自动创建
- `applications.apply` → 小程序报名
- `applications.adminCreate` → 后台代报名

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string | **报名人** → `users._id` |
| `job_id` | string | **报的岗位** → `jobs._id` |
| `company_id` | string | 企业（从岗位冗余） |
| `source` | string | `miniprogram` / `qrcode` / `admin` / `import` |
| `recommender_id` | string | **推荐人 ID**（报名时确定，固定不变） |
| `recommender_name` | string | 推荐人名称（快照） |
| `qr_code` | string | 扫码时的二维码编码 |
| `applicant_name` | string | 报名人姓名（报名时填写） |
| `applicant_phone` | string | 报名人手机号 |
| `applicant_id_card` | string | **报名人身份证号**（仅在此处存储） |
| `status` | string | `pending` → `contacted` → `interview` → `passed` / `rejected` / `cancelled` |
| `apply_time` | date | **报名时间**（日报统计用） |
| `expected_interview_time` | date | 期望面试时间 |

**关联关系：**
- `applications._id` → `interviews.application_id`（面试记录通过此 ID 关联）
- `applications.user_id` → 反查 `users` 获取候选人详情

**索引：**

| 索引字段 | 类型 | 用途 |
|----------|------|------|
| `user_id` | 必加 | "我的报名"列表 |
| `job_id` | 必加 | 岗位的报名统计 |
| `status` | 必加 | 按状态筛选（待处理/已通过） |
| `apply_time` | 必加 | **日报统计"今日报名"** |
| `recommender_id` | 建议 | 按推荐人统计报名数 |
| `service: monthly report` | 必加 | 可选年月+公司+状态筛选 |

## 👷 四、员工管理

---

### 13. employees ⭐ — 员工自然人主档

**数据来源**：办理入职时创建

**定位**：**一人一档**，不绑定任何企业。实名信息和自然人属性只在此处存储。

**核心规则：**
- `employee_no` 逻辑唯一
- 同一身份证号只允许一条有效记录
- 被合并档案不删除，设 `merged_into_employee_id`

**写入入口**：
- `employees.onboard` → 办理入职
- `employees.updateProfile` → 编辑资料
- `qrcode.internal-onboard` → 内部入职二维码

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string | **绑定的账号** → `users._id`（唯一） |
| `employee_no` | string | **员工编号**（自动生成：EP+年月日+随机4位） |
| `name` | string | **员工真实姓名** ☚ 主数据 |
| `phone` | string | **员工主手机号** ☚ 主数据 |
| `id_card` | string | **身份证号** ☚ 主数据 |
| `gender` | number | 性别 0/1 |
| `birth_date` | date | 出生日期（可由身份证解析） |
| `bank_name` | string | 开户行 |
| `bank_account` | string | **银行卡号** |
| `bank_account_name` | string | 开户名 |
| `bank_card_last4` | string | 银行卡后四位 |
| `emergency_contact` | string | 紧急联系人 |
| `emergency_phone` | string | 紧急联系人电话 |
| `is_blacklisted` | boolean | 是否黑名单 |
| `blacklist_reason` | string | 黑名单原因 |
| `merged_into_employee_id` | string | **合并后指向保留员工** → `employees._id` |
| `merge_reason` | string | 合并原因 |
| `status` | string | 兼容字段（兼容期保留，后续只做聚合缓存） |

**禁止写入**：`company_id`、`job_id`、`referrer_id`、`join_date`、`leave_date`、`hourly_rate`

**索引：**

| 索引字段 | 类型 | 用途 |
|----------|------|------|
| `employee_no` | 必加（唯一） | 员工编号查询（最核心） |
| `user_id` | 必加 | 从账号查员工（登录后获取身份） |
| `id_card` | 必加 | **身份证查重/合并** |
| `phone` | 建议 | 手机号查重 |
| `merged_into_employee_id` | 建议 | 合并档案查询 |

### 14. employee_companies 🎯 — 员工-企业在职关系

**数据来源**：办理入职时创建，每次入职一条记录

**定位**：**员工与企业之间的一段雇佣关系**。一个员工可有多条。**推荐人、企业、岗位在此处记录。**

**核心规则：**
- 写入前必须先有 `employees`
- 同一 `employee_id + company_id` 只能有一条 `active`
- 离职再入职同企业可产生新记录

**写入入口**：
- `employees.onboard` → 入职
- `employees.resign` → 离职（更新 leave_date + status）
- `qrcode.internal-onboard` → 内部入职

| 字段 | 类型 | 说明 |
|------|------|------|
| `employee_id` | string | **员工** → `employees._id` |
| `company_id` | string | **企业** → `companies._id` |
| `job_id` | string | 岗位 → `jobs._id` |
| `rate_plan_id` | string | 工价方案 → `rate_plans._id` |
| `salary_type` | string | `monthly` / `daily` / `hourly` |
| `settlement_mode` | string | `monthly` / `daily` |
| `hourly_rate` | number | 工价（优先从 rate_plan 取） |
