## 5. 自动修复候选

以下类型可以进入下一步 dry-run，不建议直接执行：

1. `users` 中 employee + candidate 且姓名、身份证、手机号高度一致的重复账号：
   - 将 candidate 侧报名、面试、推荐链路归并到 employee 账号。
   - 清理 candidate 账号中的 `id_card`、银行卡等自然人主档字段。
   - 保留审计日志。
2. 纯 candidate 重复手机号且无员工绑定、无身份证：
   - 按报名/面试/最近登录时间选择主账号。
   - 合并业务链路后标记重复账号。
3. active 关系但 `leave_date` 已过去：
   - 如果没有后续工时/薪资/保险依赖，可生成批量修复建议，把状态改为 resigned。
   - 这一步必须先做依赖检查。
4. `employees` 中关系类冗余字段：
   - 只在 `employee_companies` 对应字段完整后，才能清理 `employees.company_id/job_id/join_date/leave_date/referrer_*` 等历史字段。

## 6. 必须人工确认

以下类型不建议自动修：

1. `employees.user_id` 重复的龚诗辰两条员工主档。
2. `users.id_card=142702197305203630` 对应两个不同姓名员工。
3. 活跃员工跨多个企业关系的 2 组。
4. 手机号和身份证明显填反的员工记录。
5. 缺身份证但有薪资、保险、工时链路的员工主档。
